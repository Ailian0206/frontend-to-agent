# 生产部署与运维入门课程设计

日期：2026-07-22
状态：已完成用户评审，等待实施计划

## 1. 背景

`frontend-to-agent` 已覆盖 Agent 工程基础、实验和作品集路径，但“工程上线”仍偏概念，缺少面向前端开发者的生产平台教程。`evidence-graph` 已真实使用 Vercel、Supabase、Inngest 和 Sentry 完成生产部署，适合作为脱敏案例。

本里程碑在现有三栏学习工作台中新增完整的“生产部署与运维入门”专题。课程先解释托管生产系统，再教授日常管理和故障判断，最后引导学习者在独立练习环境完成一次从零部署。

教程不是四个平台官网的逐页翻译。它提炼个人全栈开发者最常使用的核心对象、操作和判断方法，同时保留英文界面原词、中文解释、官方来源和界面记录日期。

## 2. 目标读者与目标

### 2.1 目标读者

- 有成熟前端经验，但缺少生产后端、数据库、后台任务和可观测性经验。
- 能阅读 TypeScript 和 Next.js 项目，但阅读英文控制台和英文官方文档有障碍。
- 正在维护个人或小团队项目，需要低成本、可执行的日常运维方法。

### 2.2 学习完成标准

完成专题后，学习者应能：

1. 解释 Vercel、Supabase、Inngest、Sentry 的职责边界和协作关系。
2. 独立完成发布后、每日、每周和每月的最小巡检。
3. 看懂四个平台的核心控制台页面和常见英文状态。
4. 从用户症状选择正确的后台入口，并判断应记录、修复、回滚还是停止操作。
5. 分清只读、普通变更和高风险操作，不在生产环境完成学习性写操作。
6. 在本地或独立练习环境完成一次从 GitHub 到可观测生产环境的部署。

## 3. 范围

### 3.1 包含

- 14 课连续课程，每课目标时长 30–45 分钟。
- Vercel、Supabase、Inngest、Sentry 各两课平台中文教程。
- 平台通用教程为主体，`evidence-graph` 为脱敏真实案例。
- 约 20–28 张真实控制台完整截图或操作裁剪图。
- 关键英文菜单、状态、按钮的中文解释和用途说明。
- 发布、巡检、跨平台故障定位、安全、成本、备份和恢复准备度。
- 官方文档链接、查阅日期和平台改版后的维护规则。
- 桌面与移动端专题导航、全文搜索、学习进度和章节深链。

### 3.2 不包含

- 不逐页翻译或镜像平台官网。
- 不覆盖企业组织治理、复杂团队权限、合规采购和高级计费配置。
- 不写死会变化的套餐价格、免费额度或按钮像素位置。
- 不自动操作生产回滚、数据库恢复、迁移、密钥轮换或付费升级。
- 不在仓库保存原始控制台截图、邮箱、项目 ID、密钥、真实用户、生产正文或付费 Provider 响应。
- 不新增 CMS、内容数据库或课程运行时外部服务。

## 4. 教学主线

采用“场景主线，平台穿插”的组织方式：

1. 先建立生产系统心智模型和巡检节奏。
2. 再分别学习四个平台的常用操作和运行管理。
3. 然后把四个平台的信号串成发布与故障处理流程。
4. 最后在独立环境完成一次从零部署。

平台课采用两层内容：

- **平台通用教程**：适用于任何个人 Next.js / 全栈项目。
- **Evidence Graph 案例**：用当前真实页面展示一次脱敏只读检查和判断过程。

## 5. 课程结构

### 第一部分：生产入门

#### P01 托管生产系统地图

- 浏览器请求、Web/API、Auth、数据库、后台任务和监控的完整链路。
- 开发、Preview、Production 的差异，以及环境变量如何进入不同环境。
- 四个平台各自负责和不负责什么。
- 使用独立 `production-ops-intro` slug；Agent 课程现有 `deploy-observe` 保持原样。

#### P02 个人项目的巡检节奏

- 发布后、每日、每周、每月和故障时的检查清单。
- 正常、可疑、需要处理三类信号。
- 只读、普通变更、高风险操作三种风险等级。
- 如何记录检查结果，而不把运维变成企业级流程负担。

### 第二部分：平台中文教程

#### P03 Vercel 核心对象与常用操作

- Account / Team、Project、Deployment、Environment、Domain 的关系。
- 从 GitHub 导入项目、查看部署、定位 Build Logs 和 Runtime Logs。
- Production / Preview 环境变量与域名的常见管理入口。
- `Ready`、`Building`、`Failed`、`Canceled` 等状态的中文解释。

#### P04 Vercel 发布、日志、流量与回滚

- Git 自动部署、重新部署、Promote 和回滚的差异与风险。
- Analytics、Usage、函数日志和构建日志分别回答什么问题。
- 发布后验收、构建失败和“部署成功但页面报错”的判断路径。
- 个人项目应关注的额度与异常流量信号。

#### P05 Supabase 数据库、SQL 与迁移

- Project、Postgres、Schema、Table、SQL Editor、Migration 的关系。
- 表数据、SQL 查询、迁移历史和 API 入口的常用只读操作。
- 本地迁移到远端的正确心智模型，避免直接在生产控制台随意改表。
- 数据库健康、连接和查询异常的基本判断。

#### P06 Supabase Auth、RLS、日志与备份

- Auth User、Provider、Redirect URL、Session 和 RLS 的关系。
- 用户、Provider、Policy、Logs、Backups 和 API Keys 的常用入口。
- Publishable / anon key 与 service role key 的安全边界。
- 登录失败、权限拒绝、RLS 泄漏风险和恢复准备度。

#### P07 Inngest App、Event、Function 与 Run

- App、Event、Function、Run、Step 的关系和事件驱动流程。
- 查看已同步应用、函数、事件负载、运行步骤和错误位置。
- `Queued`、`Running`、`Completed`、`Failed`、`Cancelled` 等状态。
- 为什么长任务不应塞在一次普通 Vercel 请求里。

#### P08 Inngest 重试、并发、重放与成本

- Retry、Replay / Rerun、Cancel、Concurrency、Throttle 的差异。
- 何时可以重放，何时会造成重复副作用。
- 失败率、耗时、积压和 Usage 的日常判断。
- 幂等键、不可重试错误和成本上界。

#### P09 Sentry Issue、Event、Release 与 Source Map

- Project、Issue、Event、Release、Stack Trace 的关系。
- 从 Issue 进入单次 Event，读取环境、版本、路径和调用栈。
- Release 与 Source Map 为什么决定前端错误是否可读。
- 新问题、回归、重复问题和已解决问题的基本处理。

#### P10 Sentry Trace、告警、隐私与额度

- Trace、Transaction、Span 和性能瓶颈的基础关系。
- Alert、过滤、采样、PII 清理和配额管理。
- 如何区分偶发噪声与需要处理的错误趋势。
- Sentry 与 Vercel Logs、Supabase Logs、Inngest Runs 的互补关系。

### 第三部分：联合运维

#### P11 发布前后检查与回滚决策

- 发布前变量、迁移、回调和兼容性检查。
- 发布后版本、关键路径、任务和错误监控检查。
- 重新部署、代码修复和回滚的选择条件。
- 最小发布记录模板。

#### P12 跨平台故障定位

- 从“打不开、登录失败、数据异常、任务卡住、页面报错”进入故障树。
- 最小定位链：用户症状 → Vercel → Sentry → Supabase 或 Inngest → 决策。
- 时间、版本、请求和运行 ID 如何串联多个平台。
- 停止条件、证据记录和避免扩大故障的动作。

#### P13 密钥、权限、成本与恢复

- 客户端公开变量与服务端机密变量的边界。
- 最小权限、轮换触发条件、Usage 巡检和预算提醒。
- 数据库备份不等于恢复成功，恢复演练的最小步骤。
- 个人项目适用的周检和月检清单。

### 第四部分：综合实践

#### P14 从 GitHub 到可观测生产环境

- 使用一个独立练习项目串联 Vercel、Supabase、Inngest 和 Sentry。
- 配置环境、数据库、登录回调、后台函数和错误监控。
- 完成部署后验收、一次受控练习错误和检查记录。
- 所有写操作只在本地或独立练习环境执行；不操作 `evidence-graph` 生产数据。

## 6. 单课内容模板

平台课固定包含以下部分：

1. **本课目标**：完成后能做出的具体判断。
2. **职责边界**：平台解决什么、不解决什么、位于系统哪一层。
3. **核心对象**：用图或表解释控制台中的关键实体及关系。
4. **中文界面导览**：完整截图建立方位，关键英文附中文解释。
5. **常用操作**：裁剪截图一次只突出一个入口、状态或动作。
6. **正常与异常**：说明正常信号、异常信号和下一步。
7. **Evidence Graph 案例**：真实项目只读检查，不代替通用知识。
8. **频率与风险**：标注发布后、每日、每周、每月或故障时，以及操作风险。
9. **常见误区与停止条件**：明确何时不应继续点按钮。
10. **章节自检与官方资源**：提供可验证标准、官方链接和查阅日期。

非平台课可以按主题合并相邻部分，但必须保留判断标准、风险边界和自检。

## 7. 截图与中文标注

### 7.1 表现方式

采用用户确认的 A + B 混合方式：

- **A：完整截图 + 编号图例**。章节首次进入平台页面时保留导航方位，单图最多 5 个编号。
- **B：任务裁剪图 + 单一重点**。具体操作每图只突出一个入口、状态或判断点。
- 官网英文原词保留在图片中；中文解释位于编号图例、正文和术语表，不覆盖原界面。
- 每张图提供中文替代文字、说明、记录日期和必要的官方来源。

### 7.2 隐私与真实性

- 仅在用户明确授权后只读访问已登录控制台。
- 原始截图保存于仓库外的临时目录，完成标注后删除，不进入 Git。
- 最终图片必须隐藏邮箱、账号名称、组织或项目 ID、内部域名细节、密钥、真实用户、错误正文和生产数据。
- 密钥、账单、成员权限等敏感页面优先使用示意图，并明确标注“示意图”。
- 不把仿真界面描述为真实当前页面。
- 预计提交 20–28 张脱敏后的 WebP 图片；实际数量以覆盖判断点为准，不为凑数重复截图。

## 8. 内容与组件架构

### 8.1 内容组织

- `src/content/chapters/production-ops/`：按 overview、vercel、supabase、inngest、sentry、operations 拆分 14 课。
- `public/course/production-ops/`：只保存脱敏、标注、压缩后的最终图片。
- `src/content/resources.ts`：增加当前官方文档资源，不复制官方正文。
- `production-ops-intro` 作为 P01；Agent 课程现有 `deploy-observe` 继续保留在原课程中。

### 8.2 顶层课程归属

菜单里程碑 `feat/sidebar-dual-curriculum` 提供 `curriculum: "agent" | "production-ops"` 契约和顶层课程切换器。现有章节默认属于 Agent 课程；本里程碑的 14 课显式属于 `production-ops`。课程内 P01-P14 编号按生产课程数组顺序派生，不再新增 `series` 或二级分组模型。

专题课仍保持：

- `kind: "lesson"`
- `curriculum: "production-ops"`
- `track: "工程上线"`
- 关联现有能力 S11、E2、E3、E4，并按具体课程关联 S10。

不新增新的内容种类或能力编号。

### 8.3 标注截图块

`ContentBlock` 增加 `screenshot` 类型，至少包含：

- 最终图片路径。
- 中文替代文字。
- 图片标题或操作目标。
- 界面记录日期。
- 中文编号图例。
- 可选官方来源链接。

`AnnotatedScreenshot` 使用语义化 `figure`、`figcaption` 和有序列表渲染图片与图例。图例文本同时进入全文搜索索引。表格、步骤、警告、代码、资源和章节自检继续复用现有内容块，不增加第二套 Markdown 或 CMS 渲染链。

### 8.4 导航与路由

- 章节继续使用 `/chapter/[slug]`，不增加平行课程应用。
- 左侧顶部课程切换器提供“AI Agent / 生产运维”两个独立目录；本里程碑消费既有契约，不重复修改菜单实现。
- 生产课程内使用 P01–P14 作为视觉序号；上一课、下一课、搜索和学习进度只在当前课程范围内计算。
- 章节继续参与 sitemap；移动端目录复用相同顶层课程切换和课程内分组，不增加卡片嵌套。

## 9. 安全与错误处理

### 9.1 操作等级

- **只读**：允许在生产环境练习，例如查看部署、日志、用户数量、Run 和 Issue。
- **普通变更**：仅在本地或独立练习环境执行，例如修改变量、重放任务、调整告警和重新部署。
- **高风险**：教程只解释条件与流程；删除数据、恢复备份、轮换密钥、生产回滚和付费升级必须先备份并另行确认。

### 9.2 截图或平台不可用

- 已登录页面不可访问时，不绕过登录或权限；改用官方公开页面或明确标注的示意图。
- 平台 UI 与教程不一致时，先核对官方文档和记录日期；概念未变化只更新截图，流程变化才更新正文。
- 不在自动化测试中访问真实平台、生产 URL 或任何付费 Provider。

### 9.3 故障教学模型

每个故障案例至少回答：

1. 用户看到什么症状。
2. 第一个平台和页面是什么。
3. 哪个时间、版本、请求或运行 ID 能串联证据。
4. 什么信号排除当前平台。
5. 下一步是记录、修复、回滚还是停止操作。

## 10. 测试与验收

### 10.1 TDD 顺序

行为改动遵循 RED → GREEN → refactor：

1. 先为生产课程归属、截图块和搜索索引增加失败测试。
2. 实现最小类型、数据和渲染改动，使聚焦测试通过。
3. 再逐批增加课程正文、资源和图片，并持续运行内容契约测试。

### 10.2 自动化门禁

内容测试至少验证：

- 14 课 slug、课程数组顺序和 P01–P14 唯一且连续。
- 14 课都显式属于 `production-ops`；Agent 课程原有 `deploy-observe` 不变。
- 所有专题课都属于“工程上线”并关联有效能力 ID。
- 8 个平台课都包含平台通用内容、中文术语、截图、官方资源、风险说明和章节自检。
- 所有截图路径存在，替代文字、记录日期和图例完整。
- 截图图例文本可被课程搜索命中。
- 专题课均不是 `comingSoon`。

组件和 E2E 至少验证：

- 双课程切换、当前章节、课程内上一课 / 下一课和学习进度正确。
- `AnnotatedScreenshot` 的语义、图例和移动端布局可用。
- 搜索可通过 Vercel、Supabase、Inngest、Sentry 及关键英文术语找到课程。
- GitHub Pages 基础路径下图片和章节深链有效。

里程碑完整门禁：

```bash
GITHUB_PAGES=true npm run test:ci
npm run test:e2e
git diff --check
```

自动化测试不得调用真实或付费 Provider。

### 10.3 视觉与隐私验收

- 在 390×844、1024×768、1440×1000 检查专题导航、截图、图例和长英文术语。
- 不得出现横向溢出、文字裁切、图例重叠、图片空白或导航状态错误。
- 最终图片逐张人工检查隐私；原始图片不在 Git 状态、提交或生成报告中。
- 真实截图与示意图必须能从标题或说明中明确区分。

## 11. 实施与交付边界

这是新增完整课程能力的里程碑变更：

- 使用 `codex/production-ops-curriculum` 功能分支和隔离 worktree 实施。
- 菜单和 `curriculum` 基础契约由 `feat/sidebar-dual-curriculum` 交付；该分支合入 `main` 后，本分支通过 merge commit 同步，不修改其菜单交互。
- 按内容模型与 UI、平台教程、跨平台课程和综合验收拆成小步提交。
- 完整门禁通过后创建唯一 Draft PR。
- 按仓库规则运行一次独立 Claude 审核，修复有效问题后由最终 CI 验证，使用 merge commit 合并。

设计阶段不访问或修改生产资源。实施阶段的真实控制台访问仅用于已授权的只读截图，任何生产写操作、付费资源或敏感账号配置仍需单独确认。
