## P0 — 会让学员跑不通的问题

### P0-1 第 3 章环境搭建缺 ESM 配置，导致第 4/6/7/8 章代码全部无法运行

**位置**：`src/content/chapters.ts:277`（第 3 章 `bootstrap` 小节）

第 3 章指导学员执行 `npm init -y` + `npx tsc --init`，这产出的是 CommonJS 工程（`package.json` 无 `"type": "module"`，`tsc --init` 默认 `module: commonjs`）。但后续章节的示例都使用了顶层 `await`。

**复现结果**（隔离目录，按教程原样操作，tsx@4.23.1）：

```
ERROR: Top-level await is currently not supported with the "cjs" output format
```

**受影响范围**——课程的 4 个主线可运行实验，几乎是全部核心内容：

| 章节 | 文件 | 症状 |
|---|---|---|
| 第 4 章 | `src/weather-mail-agent.ts` | 顶层 `await agent.invoke(...)` 直接报错 |
| 第 6 章 | `src/rag-demo.ts` | 顶层 `await splitter.splitDocuments(...)` 报错 |
| 第 7 章 | `src/memory-demo.ts` | 顶层 `await agent.invoke(...)` 报错 |
| 第 8 章 | `src/supervisor.ts` | 顶层 `await graph.invoke(...)` 报错 |

**关联问题**：第 5 章测试文件使用 `import { createOrderStatusTool } from "./order-status.js"` 的 ESM 扩展名写法，同样只在 NodeNext/ESM 下成立。

**值得注意的是**：`examples/knowledge-agent` 自身配置完全正确（`"type": "module"` + `module: NodeNext` + `target: ES2022`），所以这是教程叙述与参考实现脱节，而非认知错误。

**建议修复**：把 capstone 的配置前移到第 3 章的 bootstrap 步骤。

```bash
npm init -y
npm pkg set type=module          # 新增
# tsconfig.json 显式给出，而不是依赖 npx tsc --init 的默认值：
#   "module": "NodeNext", "moduleResolution": "NodeNext",
#   "target": "ES2022", "strict": true
```

同时在第 3 章自检清单中补一条：`node --input-type=module -e "await Promise.resolve()"` 可运行。

### P0-2 移动端 Mermaid 架构图小到无法阅读

**位置**：`src/app/globals.css:777`

```css
.mermaid-canvas svg {
  width: 100%;              /* ← 根因 */
  max-width: 760px !important;
  height: auto;
}
```

`width: 100%` 强制 SVG 收缩到容器宽度，因此父级 `.mermaid-canvas` 上的 `overflow-x: auto` 永远不会触发。

**实测**：375px 视口下，一张为 760px 布局设计的 LR 流程图被压缩到约 45%，节点标签文字约 4px，完全无法辨认。由于 `flowchart` 配置了 `htmlLabels: true`，标签是 foreignObject 内的 HTML，会随 SVG 一同等比缩小。

**影响**：全站 6 张架构图（ReAct 循环、RAG 双管道、Supervisor 路由、最终项目架构等）在移动端全部失效。这些是课程的核心图解资产。

**建议修复**：

```css
@media (max-width: 760px) {
  .mermaid-canvas svg {
    width: auto;
    min-width: 560px;
    max-width: none !important;
  }
}
```

让图表在窄屏下保持可读尺寸并横向滚动。可再配合一个"← 横向滑动查看"的提示。

## P1 — 明显影响体验与可信度

### P1-1 19 个代码块零语法高亮

**位置**：`src/components/CodeBlock.tsx:56-58`

```tsx
<pre>
  <code>{code}</code>
</pre>
```

一个教编程的站点，全部代码呈现为单色白字。这是当前最影响"专业感"的单点问题。工具栏已经带了 `language` 字段但只用于展示标签。

**建议**：用 Shiki 在构建期把代码预渲染成带 token 的 HTML，存进 `chapters.ts` 或构建产物，运行时零 JS 成本 —— 这与项目现有的"静态导出 + 结构化内容"架构天然契合，也不会挤占已经吃紧的首屏预算（见 P1-4）。

### P1-2 Hash 路由 + 静态导出：全站只有 1 个可索引 URL

**位置**：`src/components/CourseApp.tsx:31-37`、`src/app/layout.tsx:17-20`

11 个章节共用 `/`，`#/chapter/rag` 这类 hash 对搜索引擎不可见。`layout.tsx` 的 metadata 是全站固定的，没有 `openGraph`、`twitter`、`canonical`、sitemap 或 robots。

**影响**：对一个以公开传播为目的的教程站，这等于放弃搜索流量；分享任意章节链接时，预览卡片始终显示首页标题。

**建议**：改为 `app/chapter/[slug]/page.tsx` + `generateStaticParams()` + `generateMetadata()`。`output: "export"` 下同样能产出 11 个独立 HTML，且能顺带解决当前深链的水合闪烁（`getServerSnapshot` 固定返回 `chapters[0].slug`，深链会先渲染第 1 章再切换）。

### P1-3 可访问性：三处实测缺陷

**位置**：`src/components/CourseApp.tsx`、`src/app/globals.css:1155`

| 问题 | 实测数据 | 影响 |
|---|---|---|
| 移动端抽屉关闭时元素仍可聚焦 | 14 个可聚焦元素，`left: -61px`，`visibility: visible`，无 `inert` | 键盘/读屏用户会 tab 进一个看不见的菜单 |
| 搜索弹窗无焦点陷阱 | 声明了 `role="dialog" aria-modal="true"`，但 Tab 可离开弹窗，关闭后不归还焦点 | 违反 dialog 语义约定 |
| 弹窗/抽屉打开时未锁背景滚动 | `getComputedStyle(document.body).overflow === "visible"` | 移动端背景穿透滚动 |

抽屉的动画方式是 `transform: translateX(-102%)`，元素仍在渲染树与 tab 序列中。

**建议修复**：

```tsx
<aside className={`course-nav ${mobileNavOpen ? "open" : ""}`} inert={!mobileNavOpen || undefined}>
```

注意 `inert` 只应在移动断点生效，桌面端侧边栏必须保持可聚焦 —— 可用 `useMediaQuery` 或改为在 `>760px` 时始终传 `undefined`。搜索弹窗补焦点陷阱与焦点归还，两处打开时给 `body` 加 `overflow: hidden`。

### P1-4 首屏 JS 257 KB (gzip)，超出项目自订预算 70%

**度量结果**（解析 `out/index.html` 引用的全部入口 chunk）：

| 项 | 数值 |
|---|---|
| 首屏 JS（gzip） | **257 KB** |
| 项目规则中的落地页预算 | 150 KB |
| CSS（gzip） | 5.5 KB ✓ |
| Mermaid 懒加载 chunk（gzip） | 142 KB（仅含图章节加载） |

主要超支来自 `react-markdown` + `remark-gfm`（完整 micromark/remark 管线），而 `ContentRenderer.tsx:18-33` 只用它渲染段落内的 `**粗体**`、行内 `code` 和链接三种情况。

内容是仓库内自有的可信结构化数据，不存在不可信 Markdown 输入，用不着完整 CommonMark 解析器。改为几十行的行内解析（或把强调/链接提升为 `ContentBlock` 的结构化字段）可省约 50–60 KB gzip。

Mermaid 已经是动态导入，这点做得对；但可再进一步用 IntersectionObserver 延迟到图表进入视口再加载。

### P1-5 Capstone 服务端违反了第 9 章自己定的验收标准

**位置**：`examples/knowledge-agent/src/server.ts:20-40`

第 9 章的示例代码和自检清单明确要求"输入有长度校验、鉴权、限流和 45 秒总超时"，且课程正文强调"无论哪种方式，都要传播 AbortSignal"。但最终项目的 `/api/chat`：

| 要求 | 第 9 章示例 | capstone 实现 |
|---|---|---|
| Zod 输入校验 | ✓ | ✓ |
| 45 秒总超时 / AbortSignal | ✓ | ✗ |
| 鉴权 | 文中要求 | ✗ |
| 限流 | 文中要求 | ✗ |
| `recursionLimit` | ✓ | ✓ |

学员照抄"最终实战项目"，学到的正好是课程刚批评过的做法。至少应补齐 AbortSignal 超时，鉴权与限流可用注释明确标注为"生产环境必须补充"并给出方向。

## 做得好的部分（建议保留）

**课程内容的工程判断力是这个项目最强的资产**，明显来自实战而非泛泛综述：

- 第 5 章"Tool 是能力契约"的坏设计对照表（`execute_any_sql` → `read_order_status`）
- 第 6 章把检索评估拆成 Retriever 层与生成层两级，并强调"正确证据没被召回时，继续改 Prompt 没有意义"
- 第 7 章记忆的"观察 / 推断 / 确认事实"三级写入策略，以及写入应比读取更保守
- 第 8 章的多 Agent 决策表，尤其是"只是想让答案更聪明 → 先改模型、上下文与评估"
- 第 11 章的失败根因分类（intent / routing / schema / integration / retrieval / generation / termination）与调试顺序

**安全意识贯穿全篇且落到了代码**：Dry Run 默认开启、幂等键、"检索内容只是数据不是指令"（capstone 的 systemPrompt 第 4 条真的写进去了）、权限在检索阶段生效而非事后让模型"不要说"。

**视觉方向明确且非模板脸**：暖纸色调 + 编辑部式排版 + 等宽字点缀，层级对比清晰。`prefers-reduced-motion` 有完整处理；320px 下实测无横向溢出（`scrollWidth === innerWidth === 320`）；代码块与表格各自在容器内滚动，做法正确。

**Mermaid 使用了 `securityLevel: "strict"`**，配合仓库内静态图表内容，此处的 `dangerouslySetInnerHTML` 是可接受的取舍。
